## 评审维度

请从以下维度逐文件审查代码改动：

### 1. 功能正确性
- 改动是否实现了预期功能
- 边界条件是否处理（空值、越界、异常等）
- 逻辑是否存在明显错误

### 2. 代码规范
- 命名是否清晰（变量、函数、类名）
- 代码格式是否符合项目规范
- 是否有冗余代码、未使用的变量/导入
- 注释是否必要且清晰

### 3. 潜在Bug
- 空指针/空值引用风险
- 资源泄漏（文件、连接、锁等未释放）
- 并发安全问题
- 类型转换风险
- 异常处理缺失

### 4. 性能影响
- 是否存在明显的性能瓶颈
- 大数据量处理是否合理
- 是否引入了不必要的循环或查询

### 5. 安全性
- 是否存在SQL注入、XSS等安全风险
- 敏感信息是否泄露
- 权限校验是否完整

### 6. 影响范围
- 改动是否会影响其他模块
- 是否有兼容性风险
- 是否需要更新相关文档

## 问题严重级别定义

| 级别 | 定义 | 处理方式 |
|------|------|---------|
| **P0** | 阻塞性bug、安全问题、会导致生产事故的改动 | 必须修复后才能合并 |
| **P1** | 功能缺陷、明显逻辑错误、性能问题 | 建议修复，如无法修复需说明原因 |
| **P2** | 代码规范、命名、格式、建议性改进 | 可选修复 |

## 产出物

调用 `workflow_advance` 时提供：

- **review_summary**：完整的评审摘要（至少30个字符），包括：
  - 改动概述（本次MR主要做了什么）
  - 逐文件/逐问题点评
  - 发现的问题列表（含位置、级别、说明、建议）
  - 整体评价（通过/有条件通过/不通过）
- **issues_found**：发现的问题数组，每个问题包含：
  - `file`：涉及文件
  - `line`：行号（如有）
  - `level`：P0/P1/P2
  - `description`：问题描述
  - `suggestion`：改进建议
- **has_p0_bug**：`true` 表示存在P0级别问题，`false` 表示没有
